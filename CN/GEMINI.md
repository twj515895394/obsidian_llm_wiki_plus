## 强制规则

### 1. 外部资料必须先判断是否触发 capture

当用户提供 URL、GitHub 仓库、本地文件路径、网络文件链接、上传文件、网页文章、论文、产品文档、API 文档或长文本资料时，必须考虑触发 `capture` Skill。

如果资料有长期价值，应主动询问用户是否保存到 `60_原始资料/`。

推荐话术：

```text
这份资料后续可能会复用，我建议保存一份到 `60_原始资料/` 并建立 source 记录，方便之后追溯和二次加工。是否保存？
```

不要在未获得用户确认的情况下，把外部资料批量复制到 Vault 中。

### 2. 原始资料和加工内容必须分离

```text
60_原始资料 = 原始证据
30_研究 = 分析过程
40_知识库 = 可复用知识
80_决策中心 = 重要判断
70_内容创作 = 对外输出
```

### 3. 复杂任务先给计划

涉及创建多个目录或文件、重组知识库结构、批量迁移笔记、对多个页面做整合、生成完整项目骨架等操作时，先输出执行计划，等待用户确认后再执行。

### 4. 优先更新已有页面，不要无脑新建

在创建新知识页、研究页、决策页之前，应先检查是否已有相关页面。

### 5. 重要结论必须保留来源

涉及技术判断、架构决策、产品判断、工具评价、模型能力判断、内容策略、数据、事实、引用、结论时，应尽量保留来源。

### 6. 不确定时进入待审核

多个资料互相矛盾、旧结论可能过期、需要删除或合并已有知识页、需要改变重要决策记录时，应将问题记录到 `90_计划/待审核/`。

## 最重要的目标

你的目标不是让 Vault 变得“文件很多”，而是让它逐步形成：

```text
可追溯的资料层

可推进的项目层
可输出的内容层
可审计的决策层
```
