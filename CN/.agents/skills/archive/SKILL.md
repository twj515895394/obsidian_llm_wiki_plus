# 归档 Skill

## 目标

帮助用户归档已完成项目、已处理收件箱条目、过期计划和阶段性资料，在保持活跃区清爽的同时保留历史记录。

本 Skill 的原则是：只归档，不删除；先确认，再移动；保留来源、链接和元数据。

## 触发条件

用户出现以下意图时使用本 Skill：

- “归档项目”
- “清理收件箱”
- “把完成的项目归档”
- “整理已处理内容”
- “这个阶段结束了，帮我收尾”
- “清理旧计划 / 旧任务”

## 可归档对象

| 对象 | 活跃位置 | 归档位置 |
|---|---|---|
| 已完成项目 | `20_项目/` | `99_系统/归档/项目/YYYY/` |
| 已处理收件箱 | `00_收件箱/` | `99_系统/归档/收件箱/YYYY/MM/` |
| 已完成计划 | `90_计划/待执行/` | `90_计划/归档/YYYY/` |
| 过期审核项 | `90_计划/待审核/` | `90_计划/归档/YYYY/` |

## 执行流程

1. 识别候选归档项
   - 查找 `status: done`、`status: processed`、`status: archived` 候选。
   - 对没有 frontmatter 的文件，基于标题、目录和用户说明判断。

2. 列出归档计划
   - 显示候选文件。
   - 显示目标归档路径。
   - 说明不会删除内容。
   - 等待用户确认。

3. 执行归档
   - 移动文件或目录到归档位置。
   - 增加或更新 frontmatter：`archived: YYYY-MM-DD`。
   - 保留原有 metadata、wikilink 和附件引用。

4. 更新记录
   - 必要时更新当天 `10_日记/YYYY-MM-DD.md`。
   - 必要时记录到 `99_系统/日志/`。

5. 输出归档报告
   - 已归档数量。
   - 目标路径。
   - 剩余待处理项。
   - 后续建议。

## 输出示例

```markdown
## 归档完成

已归档项目：
- [[项目A]] → `99_系统/归档/项目/2026/项目A.md`

已归档收件箱：
- `idea-note.md` → `99_系统/归档/收件箱/2026/05/idea-note.md`

后续建议：
- 检查是否需要为项目A生成复盘
- 处理剩余 3 条收件箱内容
```

## 边界规则

- 未经用户确认，不要批量移动文件。
- 不要删除文件。
- 不要把未完成项目归档，除非用户明确要求。
- 大项目归档前，应提醒用户是否需要先生成项目复盘。
- 如果归档会导致路径引用变化，应提醒用户检查附件和嵌入链接。
- 如果只是知识页过期，不要直接归档，应先进入 `90_计划/待审核/`。

## 质量检查

完成后检查：

- 是否列出归档计划并获得确认？
- 是否只移动不删除？
- 是否保留元数据？
- 是否添加归档日期？
- 是否生成归档报告？
- 是否提示可能需要复盘或后续处理？
