# 资料捕获 Skill

## 目标

当用户提供外部资料时，判断资料是否值得进入 Vault，并在用户确认后保存到 `60_原始资料/`，建立可追溯的 source 记录。

本 Skill 只负责“资料进入系统”，不负责深度研究和知识沉淀。后续分析应交给 `research`，后续入库应交给 `integrate`。

## 触发条件

用户输入中出现以下内容时，应检查是否触发本 Skill：

- URL / 网页链接
- GitHub 仓库或代码项目
- 本地文件路径
- 网络文件链接
- 用户上传文件
- PDF / Word / Excel / 图片 / 音频 / 视频
- 论文、官方文档、API 文档、产品文档
- 视频链接、播客、访谈、会议纪要
- 大段粘贴文本
- 用户明确说“保存资料”“收集资料”“作为后续参考”

## 资料等级

| 等级 | 含义 | 默认策略 |
|---|---|---|
| S | 长期高价值原始依据：官方文档、论文、核心 GitHub、公司资料、架构文档 | 强烈建议保存 source card，尽量保存副本或快照 |
| A | 高质量可复用资料：深度文章、竞品资料、工具文档、教程 | 建议保存 source card，可保存摘要或关键片段 |
| B | 临时参考资料：普通文章、社交帖子、一次性资料 | 询问是否保存，默认只保存链接和简要说明 |
| C | 低质量、广告、重复、明显短期资料 | 默认不保存，除非用户明确要求 |

## 执行流程

1. 识别资料类型和来源。
2. 判断资料等级：S / A / B / C。
3. 如果是 S/A/B 级，向用户说明保存价值并询问是否保存。
4. 用户确认后，创建 source card。
5. 根据类型放入对应目录：
   - 网页：`60_原始资料/来源/网页/`
   - GitHub：`60_原始资料/来源/GitHub/`
   - 文档：`60_原始资料/来源/文档/`
   - 论文：`60_原始资料/来源/论文/`
   - 视频：`60_原始资料/来源/视频/`
   - 本地文件：`60_原始资料/来源/本地文件/`
   - 粘贴文本：`60_原始资料/来源/粘贴文本/`
   - 图片/PDF/音频/视频附件：`60_原始资料/附件/`
   - 网页或仓库快照：`60_原始资料/快照/`
6. 更新 `60_原始资料/source-index.md`。
7. 如果用户还需要分析该资料，继续建议使用 `research`。

## 与 50_资源 的关系

`50_资源/` 是资源清单层，可以保存工具、链接、案例、提示词模板和轻量参考资源。

如果资料只是一个常用入口或轻量参考，可以记录到：

```text
50_资源/链接/
50_资源/工具/
50_资源/案例/
50_资源/提示词模板/
50_资源/原始资源/
```

但如果资料会作为研究、知识页或决策记录的依据，必须通过本 Skill 进入 `60_原始资料/` 并建立 source card。

不要把 `50_资源/` 当作正式证据归档层。

## 询问话术

普通资料：

```text
这份资料后续可能会复用，我建议保存一份到 `60_原始资料/` 并建立 source 记录，方便之后追溯和二次加工。是否保存？
```

高价值资料：

```text
这个资料看起来会成为后续分析的重要依据。我建议按 S/A 级资料保存到 `60_原始资料/`，并生成 source card，后续研究笔记、知识页和决策记录都可以引用它。是否保存？
```

低价值资料：

```text
这份资料看起来更像临时参考。我可以只基于它回答，也可以帮你保存一条轻量 source 记录。你希望保存吗？
```

## 输出文件

优先使用模板：

```text
99_系统/模板/source-card.md
```

推荐文件命名：

```text
YYYY-MM-DD_资料标题.md
```

示例：

```text
60_原始资料/来源/GitHub/2026-05-23_OrbitOS-CN.md
```

## source-index 更新格式

在 `60_原始资料/source-index.md` 中追加：

```markdown
- [[来源/GitHub/2026-05-23_OrbitOS-CN]]：Obsidian + Agent 工作流项目，等级 A，已用于 OrbitOS 改造研究。
```

## 边界规则

- 不要在用户未确认时批量保存外部资料。
- 不要把 AI 总结当作原始资料。
- 不要覆盖原始文件、原始链接或原始文本。
- 如果只能保存链接，必须说明没有保存完整副本。
- 如果只是工具、导航或案例入口，可放入 `50_资源/`；如果作为证据，必须放入 `60_原始资料/`。
- 如果资料涉及隐私、敏感或公司内部信息，必须提醒用户确认保存范围。
- 如果资料后续需要深度分析，保存完成后交给 `research`。

## 质量检查

完成后检查：

- 是否记录了原始来源？
- 是否写清保存原因？
- 是否标记了资料等级？
- 是否放入正确子目录？
- 是否更新 `source-index.md`？
- 是否说明了后续用途？
