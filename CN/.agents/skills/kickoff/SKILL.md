# 项目启动 Skill

## 目标

帮助用户启动一个新项目、新系统、新专题或长期任务，并在 `20_项目/` 中创建可推进、可复盘、可沉淀的项目文档。

项目启动不只是创建一个文件，而是建立“目标、范围、计划、风险、决策、资料、交付物”的工作空间。

## 触发条件

用户出现以下意图时使用本 Skill：

- “我要做一个项目”
- “帮我设计一个系统”
- “启动一个专题”
- “规划一下这个方案”
- “我们一步步做……”
- “生成项目骨架”
- “把这个想法变成可执行项目”

## 执行流程

1. 澄清项目目标
   - 项目要解决什么问题？
   - 目标用户或使用者是谁？
   - 最终交付物是什么？
   - 成功标准是什么？

2. 确定范围
   - 本期做什么？
   - 本期不做什么？
   - 哪些内容后续再做？

3. 识别资料和依赖
   - 是否已有外部资料需要 `capture`？
   - 是否需要先做 `research`？
   - 是否已有相关知识页或决策记录？

4. 生成启动计划
   - 阶段拆分
   - 关键任务
   - 风险点
   - 需要用户确认的问题

5. 等待用户确认
   - 批量创建项目文档前必须确认。

6. 创建项目文档
   - 默认使用 `99_系统/模板/project-note.md`。
   - 保存到 `20_项目/`。

7. 建立关联
   - 相关资料：`60_原始资料/`
   - 相关研究：`30_研究/`
   - 相关知识：`40_知识库/`
   - 相关决策：`80_决策中心/`
   - 计划与待办：`90_计划/`

## 输出位置

默认：

```text
20_项目/
```

推荐命名：

```text
YYYY-MM-DD_项目名称.md
```

如项目较大，可创建目录：

```text
20_项目/项目名称/
├── README.md
├── 计划.md
├── 资料.md
├── 决策.md
└── 复盘.md
```

## 项目文档结构

```markdown
# 项目名称

## 项目背景

## 项目目标

## 范围

## 交付物

## 阶段计划

## 当前状态

## 关键资料

## 关键决策

## 风险与阻塞

## 复盘记录
```

## 边界规则

- 不要在目标不清晰时直接创建复杂项目结构。
- 大项目先给计划，确认后再批量建文件。
- 项目中的重要判断应同步到 `80_决策中心/`。
- 项目中形成的方法论应通过 `integrate` 沉淀到 `40_知识库/`。
- 项目相关原始资料应通过 `capture` 进入 `60_原始资料/`。

## 质量检查

完成后检查：

- 项目目标是否明确？
- 是否写清范围和不做事项？
- 是否有阶段计划？
- 是否标记关键资料和决策？
- 是否有风险与阻塞？
- 是否说明下一步行动？
